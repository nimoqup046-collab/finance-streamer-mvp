# Claude Core Baseline

## 目的

这份文档用于固定 `finance-streamer-mvp` 当前的内容生产主线，避免后续继续在多套提示词、多条功能分支之间来回摇摆。

从现在开始，后续内容质量优化、提示词升级、风格打磨，统一以 **Claude 最近几轮高价值 PR** 为核心基底推进。

## 当前主线结论

当前仓库应当以以下 PR 作为核心脉络理解：

1. `PR #6`
   链接：<https://github.com/nimoqup046-collab/finance-streamer-mvp/pull/6>
   作用：前端交互与可用性升级基底。

2. `PR #7`
   链接：<https://github.com/nimoqup046-collab/finance-streamer-mvp/pull/7>
   作用：Claude 内容提示词系统基底，是后续内容质量优化的核心来源。

3. `PR #8`
   链接：<https://github.com/nimoqup046-collab/finance-streamer-mvp/pull/8>
   作用：把 `PR #7` 的 Claude 提示词系统，与当前工程增强能力正式整合到主线。

## 各 PR 职责划分

### PR #6：前端产品体验基底

`PR #6` 的价值不在“内容质量”，而在“这个工具是否顺手、是否像一个能持续使用的产品”。

它确立了以下前端能力：

- 生成参数面板
- 生成进度步骤反馈
- 搜索与高亮
- 收藏
- 已选新闻摘要
- 在线编辑
- 字数统计与阅读时间
- 历史记录
- 结果 Tab 切换
- 富文本复制
- 快捷键
- Toast 分级提示

后续前端优化默认原则：
- 优先在 `PR #6` 的交互骨架上继续增强
- 不回退到更早的简陋交互模式
- 内容功能新增时，尽量接入已有的历史、Tab、Toast、设置持久化体系

### PR #7：内容生产系统基底

`PR #7` 是当前内容质量路线的真正核心。

它带来的不是“多一个按钮”，而是整套内容生产方法论：

- 多 system prompt 分工
- 直播稿 / 公众号 / 深度长文 / PPT 脚本分轨生成
- 两步深度生成法
- 内部分析框架
- 更强的商业洞察、结构化表达、讲者视角
- 更适合中文财经内容的提示词约束

后续所有内容质量相关优化，应默认以它为基准：

- 优化提示词：在 Claude 基底上改
- 提升风格：在 Claude 基底上改
- 加强洞察：在 Claude 基底上改
- 调模型适配：围绕 Claude 提示词适配智谱/其他模型

不建议再新开另一套独立提示词系统与之并存。

### PR #8：Claude 基底与工程能力整合层

`PR #8` 的职责是把 `PR #7` 的内容内核，接回到当前已经做好的产品工程里。

它固定了以下整合原则：

- Claude 提示词系统作为唯一内容生产主基底
- 智谱作为默认推荐模型供应商
- 保留热点排序、更多新闻源、100 条新闻抓取
- 保留 SSE 流式生成
- 保留 `.pptx` 文件下载
- 将文本型 PPT 内容统一命名为 `ppt_script`
- 将二进制 PPT 下载保留在 `/api/generate/ppt`

这使项目形成了清晰的双层结构：

1. 内容内核：Claude 提示词系统
2. 工程载体：抓取、排序、SSE、PPT 下载、Railway、前端交互

## 当前仓库的推荐理解方式

后续维护时，请按下面顺序理解项目：

1. **内容基底先看 `backend/generator.py`**
   这里是 Claude 主提示词系统和内容生成策略。

2. **接口编排看 `backend/main.py`**
   这里负责把生成类型、SSE、批量生成、PPT 下载组织成产品接口。

3. **新闻质量看 `backend/fetcher.py`**
   这里决定新闻源、去重、热点排序、数量上限。

4. **产品体验看 `frontend/app.js` + `frontend/index.html` + `frontend/style.css`**
   这里承接 PR #6 的交互增强，以及 PR #8 的 `ppt_script` / 流式整合。

## 当前推荐的后续开发原则

### 一、内容质量优先级最高

如果后续要继续提升工具价值，优先级顺序建议固定为：

1. 提升直播稿质量
2. 提升公众号文章质量
3. 提升深度长文质量
4. 提升 PPT 脚本与 `.pptx` 的实际复用价值
5. 最后再做 GUI 美化

### 二、提示词只保留一套主系统

允许做增量优化，但不再建议：

- 同时保留两套完全独立的内容提示词系统
- 在 `generator.py` 中继续叠加相互竞争的 prompt 分支
- 用 provider 条件分支去写多套完全不同的内容逻辑

推荐方式：

- Claude 提示词结构保持为主
- 智谱 / 其他模型只做“适配层优化”
- 内部 `editorial_brief` 这类补强逻辑只作为内部增强，不单独成为第二套体系

### 三、PPT 保持双轨并存

当前 PPT 相关功能必须区分为两个产物：

1. `ppt_script`
   含义：文本型 PPT 演讲脚本 / 逐页说明稿

2. `/api/generate/ppt`
   含义：下载 `.pptx` 文件

后续不得再把这两者混用成同一个语义，否则前后端会再次冲突。

### 四、未来新功能的落点建议

如果后续增加新能力，优先往以下方向投放：

- 更强的新闻筛选与热点洞察
- 更高质量的中文财经写作风格
- 更强的生成后编辑与复用能力
- 更完整的直播工作流支持

而不是优先投到：

- 花哨但无实际增益的 UI 动效
- 第二套独立生成内核
- 破坏现有接口语义的重命名

## 当前应忽略的旧路线

以下路线已经被主线吸收，后续不应作为新的开发起点：

- 早期 Copilot 的独立功能 PR（例如 `PR #5`）
- 早期未整合的简化版 prompt 写法
- 把 `ppt` 同时当作文本结果和文件下载的旧语义

这些内容的有价值部分已经被并入或升级，保留它们只会让仓库理解变乱。

## 一句话基线

**以后这个项目的内容生产系统，以 `PR #7` 为内容基底，以 `PR #8` 为工程整合基底，以 `PR #6` 为产品交互基底。**
